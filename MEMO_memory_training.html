Training memory into a small LLM — and proving it isn’t knowledge or skill

Project memo · Qwen3-1.7B · Fireworks Reinforcement Fine-Tuning (GRPO) · task derived from CLBench blind_spectrum_monitoring · July 2026 · repo: sr-networks/clbench-fireworks-rft

Headline. Reinforcement learning (GRPO) reproducibly improves the model’s memory recall — the fraction of its remembered state it carries forward each step — from ≈0.89 to ≈0.94 in two independent runs, on task content that never repeats (so nothing can be memorised into the weights), against a scrambled-memory control that stayed flat (so non-memory skill gains are excluded). The effect is small at the score level, and it does not beat what one paragraph of explicit instruction installs for free: at this model size, prompting ≻ RL for memory use — and RL actively destroys memory the model must maintain itself.

1 · Background: the benchmark we start from

CLBench tests whether an agent learns during a task: it runs a model through a long series of task instances and measures whether late instances are handled better than early ones. Its systems differ only in how memory is provided:

CLBench systemDefinition (from source)
icl“Accumulates all interactions in a linear context window with token-based truncation from the start (FIFO)… maintains a conversation history as a list of messages… observations are prepended to the next query so the LLM sees feedback in context.” — i.e. full-history replay; nothing is summarised, the oldest turns fall off when the context overflows.
icl_notepad“ICL system with notepad… the model can maintain a notepad to jot down important observations, patterns and thoughts that persist across interactions.” — the model re-writes a free-text notepad each turn; the notepad persists, conversation may be cleared between instances.

The task (blind_spectrum_monitoring): a hidden radio band contains ~11–14 transmitters (8 MHz each in 0–180 MHz). Each scan reveals only the ~3 currently-active transmitters, as noisy detector peaks (miss rate 0.15, false-alarm rate 0.2). The rest are dormant — still occupying the band, just silent this scan. The agent must report all occupied regions; dormant transmitters can only come from memory. CLBench’s official schedule: 90 instances across 3 stages, scored by available-spectrum IoU.

2 · Our adaptation for RL training

We turn one session into one RL rollout: 12 scans of one hidden band. The model answers each scan with a submit_report(center_freqs) tool call; the environment scores it and delivers the next scan.

2.1 The metrics, defined

Both metrics score the same report against the same hidden ground truth; they differ in which side of the band they compare. Let the band be the interval [0, 180] MHz and measure set sizes in MHz:

  • True occupied O* = the union of all transmitters’ 8-MHz intervals — dormant ones included (~12 × 8 ≈ 96 MHz). True available A* = band \ O* (≈ 84 MHz).
  • A report of center frequencies implies claimed occupied O = the union of 8-MHz intervals at the reported centers, and by complement claimed available A = band \ O.
occupied-IoU  =  |O ∩ O*| / |O ∪ O*|          available-IoU  =  |A ∩ A*| / |A ∪ A*|

Fig 0 — the two metrics on one example. A memoryless report (3 current peaks, 24 MHz) against a 12-transmitter ground truth. Occupied-IoU compares the thin report to the full transmitter map: 24/96 ≈ 0.25 (detector noise → ≈0.16). Available-IoU compares the complements: the true free space (84 MHz) is almost entirely inside the claimed free space (156 MHz), giving 84/156 ≈ 0.54 despite total amnesia — each forgotten dormant transmitter costs 8 MHz out of a ~96-MHz target on the occupied side, but only nudges a ~156-MHz union on the available side. That asymmetry is the entire reason we train on occupied-IoU.

Reward = 3 × mean occupied-IoU over the rollout. memory_gain = (occ of late half) − (occ of early half) is measured but never rewarded — rewarding a delta invites sandbagging (tank the early scans, “improve” later).

2.2 The memory channel (“running-list echo”)

The context is windowed to the current scan (the model cannot re-read earlier scans — bench-ICL is switched off). Instead, every scan carries an echo of the model’s own previous answer:

=== YOUR RUNNING LIST (your previous report) ===
19.8, 42.2, 52.4, 138.9, 150.7
Keep ALL of these, add any NEW peaks from this scan, and submit the FULL updated list.

This is a notepad with the write-policy hard-wired to “notepad := my last answer”: storage and retrieval are free, only use (keep + merge) is the model’s job. That isolates the one competency we measure. Where the model must also maintain the memory (notepad tools), see §9.

3 · The threat model: what could fake “memory training”

confoundhow it fakes memoryour counter-measure
knowledge (content baking)the same band layouts recur across training; the model memorises them in its weights and “recalls” without memoryepoch-salting: a fresh random band set every epoch — no band is ever seen twice; every epoch is automatically a fresh-content evaluation. (This mattered: all our pre-rigor runs had unknowingly reused the same 48 bands.)
skill (non-memory mechanics)RL improves JSON formatting, peak transcription, verbosity — score rises without memoryscrambled-memory control arm: identical training, but the echoed list is replaced by random frequencies (same count/format/instruction). Whatever survives scrambling is skill.
luck / cherry-pickinga single noisy curve is declared a trendtrend = halves + slope + R², plus an independent replication run; behaviour measured directly from traces (carry-rate = fraction of the echoed list preserved in the next report).

GRPO detail that makes any of this work: the 12 reward-compared candidates of a dataset row must face the same band (seeded per row), otherwise advantages measure band luck, not policy — with fully random bands the policy provably never moves (our first six runs were flat for exactly this reason).

5 · The prompts (verbatim)

5.1 Explicit scaffold prompt (arms A, B) — teaches accumulation

You are a spectrum-monitoring analyst watching ONE fixed radio-frequency band over a SERIES of scans, shown ONE AT A TIME. Each scan lists noisy detected peaks (frequency, power, width).

KEY FACT: transmitters do NOT go away. A transmitter that is silent (dormant) in one scan is STILL occupying the band and will often reappear later. You are scored on how well your report matches the band's TRUE occupied spectrum, which includes every dormant transmitter — so report EVERY transmitter you have ever seen, not just the ones detected in the current scan.

You cannot look back at earlier scans. To help you, each scan includes YOUR RUNNING LIST — the report you submitted for the previous scan. Keep ALL of it, add any NEW peaks from the current scan, and submit the FULL updated list via submit_report(center_freqs) (each region is a fixed 8 MHz wide; it returns the next scan).

Think BRIEFLY, then act — keep your reasoning to a sentence or two, do not over-explain.

5.2 Weak scaffold prompt (arms C, C′) — task only, accumulation NOT taught

You are a spectrum-monitoring analyst watching ONE fixed radio-frequency band over a SERIES of scans, shown ONE AT A TIME. Each scan lists noisy detected peaks (frequency, power, width). Each scan, report the center frequencies (MHz) of the transmitter regions that occupy the band via submit_report(center_freqs) (each region is a fixed 8 MHz wide; it returns the next scan). You are scored on how well your reported regions match the band's true occupied spectrum.

You cannot look back at earlier scans. Each scan includes YOUR RUNNING LIST — the report you submitted for the previous scan.

Think BRIEFLY, then act — keep your reasoning to a sentence or two, do not over-explain.

The only remaining nudge in the weak arms is the echo block’s own “keep all + add new” line (§2.2), which is part of the channel itself. The difference A−C at epoch 0 (0.472 vs 0.403) is therefore what the explicit paragraph buys.

5.3 The earlier notepad-tools prompt (for §9), deliberately vague variant

You are a spectrum-monitoring analyst watching ONE fixed radio-frequency band over a SERIES of scans, shown ONE AT A TIME. Each scan lists noisy detected peaks (frequency, power, width).

KEY FACT: transmitters do NOT go away. A transmitter that is silent (dormant) in one scan is STILL occupying the band — it just wasn't detected this scan, and it will often reappear later. Only some transmitters are active in any given scan; the rest are dormant but STILL PRESENT. You are scored on how well your report matches the band's TRUE occupied spectrum, which includes every dormant transmitter. So you must report EVERY transmitter you have …

Note the mechanics: the next scan arrives as a user turn; the echo carries the previous answer; SCAN_OCC is the per-scan score the reward parser reads (windowed away from later decisions). “Memory use” is visible as the report growing while old frequencies persist. The carry-rate instrument counts exactly that: what fraction of the echoed list survives into the next report.

7 · Results

Fig 1 — mean occupied-IoU per epoch, all four arms (bands never repeat across epochs). A flat at the instructed level; B flat (control); C and C′ drift upward from the weak base toward A.

Fig 2 — the behavioural instrument: carry-rate (recall) at epoch 0 vs epoch 11. The weak arms improve recall by +0.05 / +0.035, converging to ≈0.94; the explicit arm starts saturated (0.967).

Finding 1 — RL trains memory recall (C + C′)

From the weak base, GRPO improved recall in both independent runs (carry 0.888→0.938, 0.900→0.935; report size 12.0→14.5 / 12.9→13.5; occ slopes +0.0037 and +0.0019/ep, both runs ending at their maximum). On never-repeating bands with the scrambled arm flat, the improvement can only be policy-level memory use. Honest weights: the score-level effect is small (+0.012–0.022 by halves vs ±0.02 band noise — it clears the bar via slope consistency and the behavioural measure, not magnitude); memory_gain rose in C but not C′ (it is the noisiest instrument); 12 epochs did not reach the instruction ceiling.

Finding 2 — the causal chain, measured at every link

Within a GRPO group (same band): candidates differ behaviourally (sd(carry)=0.073) → reward ranks that behaviour (corr(carry, reward) = +0.75…+0.80 — the reward design was never the bottleneck) → the policy moves along the paid gradient (carry rises).

Finding 3 — instruction beats RL at this scale (A + saturation diagnosis)

The explicit paragraph alone installs occ 0.452–0.474 — above every RL-trained level in this project and above the naive accumulate-everything oracle (0.447): the instructed model already curates (filters false alarms). Its carry is 0.967 with reports at the 16-slot cap: saturated. Arm A’s flat curve means “no headroom”, not “RL can’t learn” — the sliver that existed, RL took (0.967→0.972).

Finding 4 — RL preserves cheap memory, erodes expensive memory (§9)

With the env maintaining the channel, RL held (A) or improved (C/C′) the behaviour. With memory as the agent’s own job (notepad tools), the same optimizer eroded it (20 epochs: gain +0.055 → −0.008, occ → memoryless floor): when memory upkeep costs actions, the policy drifts to the cheaper memoryless strategy.

Detector noise compresses everything: a real agent’s ceiling is ≈0.50–0.55. The instructed model has already banked most of the recall value plus some curation; the remaining headroom is in curation subtleties (drop never-reseen entries, average sighting positions).

9 · The earlier chapter: agent-managed notepad (why “use” vs “maintenance” matters)

Before the echo design we gave the model actual memory tools — notepad_read() / notepad_write(text) — with context windowed per scan, so the notepad was its only memory.

  • Vague prompt (“memory is useful, decide what to store”): the model never explores accumulation — it treats every scan as an independent question. Real trace (thinking model):
[user]   --- Scan 3/12 --- ...  (notepad tools available: notepad_read / notepad_write)
[assistant·think]  "... the current scan is the third scan, but the user hasn't provided any previous data.
                    So the notepad is empty initially. So I can just process this scan without any prior
                    data. ... the detected peak is at 84.5 MHz ... call submit_report with [84.5]."
[assistant]  submit_report({"center_freqs": [84.5]})     <-- reports ONLY the current scan, never writes
[tool]   ok   SCAN_OCC: 0.0574                                the notepad; occ stays at the memoryless floor
  • Result: 10 epochs at the memoryless floor (occ ≈0.16, gain ≈0) in two rollout interfaces, thinking on and off — an exploration failure: GRPO never sees an accumulating rollout it could reinforce.
  • Explicit prompt unlocks the notepad instantly (occ ≈0.28, gain +0.05, reproduced 3×) — but RL never improved it, and a 20-epoch run destroyed it (gain +0.055 → −0.008). Cutting corners on memory upkeep is the locally-cheaper policy, and prolonged optimisation finds it.

11 · Limitations

  • One model (Qwen3-1.7B), one task family, one optimizer (GRPO via Fireworks RFT). lr/temperature/group-size were not swept — justified now only for pushing magnitude, since signal and headroom are verified.
  • Score effects are small against band noise (±0.02); the claim leans on the behavioural instrument (carry) and slope consistency, and memory_gain replicated in only one of the two C-runs.
  • Arms draw independent band sets per epoch (the platform exposes no epoch id for paired draws).
  • The full-history (bench-icl) variant of the early positive result predates the fresh-band rigor: confounded, unrefuted.
  • RL never beat the instruction ceiling — the honest ordering at 1.7B is instruction ≻ RL ≻ nothing.
