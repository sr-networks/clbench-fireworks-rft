Can reinforcement learning teach a small LLM to use memory?
A controlled study on the CLBench continual-learning benchmark

July 2026 · Qwen3-1.7B · GRPO via Fireworks Reinforcement Fine-Tuning · task, metric and protocol: CLBench blind_spectrum_monitoring, used 1:1 · repo sr-networks/clbench-fireworks-rft

Abstract. We ask whether reinforcement learning can teach a small language model (Qwen3-1.7B) to use memory during a long task, under the strictest possible protocol: the CLBench benchmark task unmodified, scored by its own metric, with the system prompt as the only experimental variable. We trained four arms (2 memory mechanisms × 2 prompts, 8 epochs GRPO each) and evaluated on the official 90-instance schedule. Result: a complete, well-characterized null. Every training curve is a flatline at the no-memory baseline; the final benchmark scores of trained models equal the untrained ones (≈0.21–0.23, vs GPT-5.4's 0.35–0.62). Trace-level analysis localizes the failure to a single missing operation — merging new observations into the remembered set — and a quantitative gradient diagnosis shows why RL cannot fix it here: the benchmark's native metric pays ≈0.001 per remembered item, leaving the learning signal ~5× below noise. The result is not "RL cannot teach memory": in a companion study on the same model and optimizer, a densified reward (paying per remembered item) reproducibly trained recall. The binding constraint is reward density, not model scale alone.

1 · The benchmark, in plain words

CLBench measures whether an agent gets better while it works. Picture a security guard watching a large parking garage through a peephole that shows a few cars at a time. Cars park for hours but their engines run only intermittently — each "scan" reveals only the 2–4 currently active ones. After every scan the guard must report every occupied spot, including cars currently silent. A guard with no memory reports only what is visible right now and scores the same forever; a guard who keeps track improves scan after scan. The benchmark plots the per-scan score over a 90-scan session: flat = no learning, rising = learning during the task.

(The actual task is radio-spectrum monitoring — transmitters on a fixed frequency band, a noisy detector that misses 15% of active signals and hallucinates ~20% phantom peaks, reports = center frequency + width of every occupied region — but nothing below requires the radio details.)

CLBench compares memory systems on this task:

  • stateless — no memory at all: the floor. Site reference: ≈0.22, flat.
  • ICL — the whole conversation stays in context; the model can re-read every earlier scan. GPT-5.4 reference: 0.22 → 0.62 over a session.
  • ICL-notepad — context is cleared between scans; the model maintains a small text notepad (rewriting it each turn), which is the only thing that persists. GPT-5.4 reference: 0.22 → 0.35.

The score is the benchmark's own per-instance reward (an intersection-over-union of the implied free spectrum, 0–1). Two properties matter later: (i) frontier models earn large in-session gains — the task is solvable; (ii) the metric pays mostly for a complete and precise picture — partial memory earns almost nothing (measured below).

2 · Question and design

Question: a 1.7B model shows none of this in-session learning out of the box. Can GRPO-based reinforcement learning instill it — and does a prompt that tells the model to use its memory change what RL can do?

Protocol constraints (fixed a priori): the CLBench task bit-for-bit (official band configurations, official scan generator and prompts, official response format including region widths), the benchmark's own reward as the RL objective, and only the system prompt varies between arms. Each training rollout is one official 30-scan series; reward = mean per-scan benchmark score (a constant ×3 scaling cancels inside GRPO). 24 rollouts × 12 GRPO candidates per epoch, 8 epochs, lr 1e-4, Qwen3-1.7B.

ICL arms — shared task text

You are a spectrum-monitoring analyst working through a SERIES of scans of one radio band. Each scan lists noisy detected peaks (frequency, power, width). After each scan, submit your occupancy report with the submit_report tool: center_freqs = the center frequency (MHz) of every region you believe is occupied, and bandwidths = the width (MHz) of each region, in the same order. You are scored on how well your report matches the band's true occupancy. Think BRIEFLY, then act — keep your reasoning to a sentence or two.

…plus the nudge (ICL-nudge only)

IMPORTANT — use your memory: this is ONE fixed band across all scans, and transmitters that are silent in the current scan are usually still there (they transmit intermittently). Your conversation history contains every earlier scan and report — re-read it, keep track of every transmitter you have EVER seen with its observed width, and report ALL of them every scan, not just the ones detected now.

Notepad arms — shared interface text

You are a spectrum-monitoring analyst working through a SERIES of scans of one radio band, shown ONE AT A TIME. Each scan lists noisy detected peaks (frequency, power, width). After each scan, submit your occupancy report with the submit_report tool: center_freqs = the center frequency (MHz) of every region you believe is occupied, and bandwidths = the width (MHz) of each region, in the same order. You are scored on how well your report matches the band's true occupancy.

You CANNOT see earlier scans or your earlier reports. Each scan shows YOUR NOTEPAD — a text that persists from scan to scan. You may replace it by including the optional notepad_update field in your submit_report call; otherwise it stays unchanged. The notepad is the only thing that persists.

Think BRIEFLY, then act — keep your reasoning to a sentence or two.

…plus the nudge (NP-nudge only)

IMPORTANT — use your notepad as memory: this is ONE fixed band, and transmitters that are silent in the current scan are usually still there (they transmit intermittently). Keep a complete list in your notepad of every transmitter you have EVER seen, with its observed width, and update it every scan. Report ALL transmitters on your notepad every scan, not just the ones detected now.

5 · Why — the failure, localized and quantified

5.1 The missing operation is the merge

Reading the trained notepad arm's rollouts (288 rollouts, epoch 6): the model writes its notepad on 95% of scans, in exactly the requested format (frequencies + widths), and reads it back every scan. But the notepad holds 2.5 entries early → 2.8 entries late, against ~13 transmitters to remember: each turn it overwrites the notepad with roughly the current scan, discarding the rest, despite the prompt explicitly demanding the full historical list. Storage ✓, retrieval ✓ — curation ✗. The single operation "new notepad = old notepad ∪ new observations" never happens. (The ICL arms fail one step earlier: with the full transcript available, the model simply doesn't mine it.)

5.2 Why RL can't fix it here: the gradient is real but ~5× below noise

GRPO learns from score differences among 12 candidate rollouts that face identical scans. Measured on those groups (trained notepad arm):

The root cause is the metric's payment schedule. On this densely occupied band, remembering one additional transmitter is worth ≈+0.001 on the benchmark score (all naive memory together: +0.012), because the metric grades the map of free spectrum — nearly right even with total amnesia. Large payments (0.22→0.35→0.62) unlock only after a package: a near-complete list and accurate widths and false-alarm filtering. The current policy's candidates all live on the flat part of that payoff curve, so the advantage signal is ~0.005 against ~0.009 of scan noise. Policy gradient cannot cross a flat valley on breadcrumbs — 8 epochs measurably change nothing, and 10× the compute would be needed to even try.

6 · The contrast that completes the picture

In a companion study (same model, same optimizer, same task family; see RESULTS_memory.md / the project memo) we replaced the metric with one that pays per remembered item (occupied-spectrum IoU: ≈+0.03–0.08 per transmitter, paid immediately), randomized the band every epoch so nothing could be memorized into weights, and added a scrambled-memory control arm. There, GRPO reproducibly improved recall — the fraction of remembered items carried forward rose 0.89→0.94 in two independent runs, with the control flat. Same 1.7B, same GRPO:

  • dense reward (pays per item) → memory use trains;
  • the benchmark's native reward (pays for the finished package) → nothing trains.
Main conclusion. For small models, whether RL can teach memory-use is decided less by the model or the optimizer than by where the reward puts its money. CLBench's native metric is a fair examination — frontier models pass it — but it is a poor teacher: it grades the finished skill, not the steps toward it. RL training needs the steps priced individually.

7 · Honest limitations

  • One model (Qwen3-1.7B), one optimizer (GRPO/RFT), one task family; 8 epochs and lr 1e-4 per arm (no hyperparameter sweep — justified only now that the signal diagnosis shows what a sweep would be fighting).
  • Final evaluation: 3 runs per cell; run-to-run variance on the diagnostic metric is substantial (one trained-notepad run reached 0.257/0.429 — sporadic real memory episodes exist).
  • The ICL-neutral arm required four launches for infrastructure reasons (deployment cold-start races; a response-dependent crash) — fixed in code; its final training run is clean.
  • GPT-5.4 reference numbers are read from the public CLBench site, not reproduced by us.
  • "Memory" here = the strict sense (model-maintained notepad); full-history ICL is reported separately throughout.
